# 9. Technical Stack & Roadmap

## 9.1 Current Stack (M1 Prototype)

| Property | Detail |
|---|---|
| Engine | Phaser.js 3.60 — single HTML file, runs in any browser, no install |
| Language | JavaScript (ES6+) |
| Canvas Size | 960 × 600 px, FIT scale mode |
| Scenes | BootScene → ProposalReviewScene → GameScene → CycleEndScene → (YearEndScene) |
| State Passing | Via Phaser `scene.start()` data payload |
| Dev Stats Storage | `localStorage['tbf_stats']` — cycle stats auto-appended each Cycle End; exportable as JSON |
| Proc. Gen. | Weighted random tables (APPS × TECHS × sample count) |
| Distribution | Single .html file; shareable by URL or download |

## 9.2 Dev Tooling

| Tool | File | Purpose |
|---|---|---|
| Balance Calculator | `BalanceCalculator.html` | Interactive sliders for economy (funding, rep), proposal math, and throughput analysis |
| DES Engine | (inside BalanceCalculator) | Discrete Event Simulation of the full player pipeline — models 9 sample stages, station queuing, proposal queueing; runs 30–60 cycles per config to give P10/median/P90 throughput estimates |
| Calibration | Throughput tab → Calibration section | Enter observed samples/cycle and proposals/cycle from real play; tool computes model error and suggests walk-time adjustments |
| Stat Export | CycleEnd dev buttons | Each Cycle End auto-saves a JSON record to `localStorage`; **Export Stats JSON** downloads all records; **Clear Stats** resets |
| Dev Panel | `BrilliantFacility.html?dev` | Overlay shown during gameplay when `?dev` is in the URL. Buttons: `+$100k` (add funding), `+20 ⭐` (add rep), `+1 Cy` (skip to next cycle), `+1 Yr` (skip to next year + apply grant), `1×` (cycle time speed: 1×, 2×, 4×, 8×) |
| AI Playtesting Bot | `js/bot.js` | Automated player for design testing. Always loaded; activate via browser console. See below. |

### AI Playtesting Bot

The bot (`js/bot.js`) takes over player control to stress-test game balance without manual play. It hooks into Phaser's `postupdate` event, overrides `px`/`py` each frame, and reuses the scene's existing `_pathCrossesRing` / `_orbitRing` helpers for navigation.

**Console API**

| Command | Effect |
|---------|--------|
| `_bot.start()` | Bot takes over; keyboard suppressed |
| `_bot.stop()` | Restore keyboard control |
| `_bot.stats()` | Print cycle metrics as a console table |
| `_bot.speed(n)` | Set `devTickMult` (requires `?dev`) |

**Priority order** (re-evaluated every 250 ms):

1. Collect any finished measurement results
2. Stay in control room while a measurement is running
3. Start a deposited-but-unstarted measurement
4. Deposit `exp_setup_done` sample at control room
5. Stay near hutch during experiment setup
6. Trigger experiment setup at the correct beamline
7. Pick up finished prep samples
8. Deposit raw samples at the correct prep table (waits nearby if full, keeping the proximity gate active)
9. Collect raw samples from NPCs (prioritises users with the shortest time remaining)

Proposals are accepted automatically every tick with no movement cost.

**Design use**: watch the bot's overlay label over a full cycle. Time spent on goals 2–3 (`wait meas` / `start meas`) indicates the measurement phase is the throughput bottleneck. Time spent on goal 8 (`wait wet/dry lab`) indicates prep capacity is the bottleneck.
